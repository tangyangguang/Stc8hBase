# 芯片支持范围

## 1. 支持原则

第一版以 STC8H1K08 为主目标芯片。

在不增加代码复杂度、不引入运行期适配层、不牺牲未使用模块零占用原则的前提下，尽量兼容同系列、同寄存器模型、同外设行为的 STC8H 芯片。

## 2. 支持等级

### 2.1 主支持芯片

主支持芯片是：

- `STC8H1K08`

主支持芯片要求：

- 基础库模块以它的资源和寄存器模型为第一设计目标。
- 示例优先在它上面验证。
- 默认配置优先满足它的常用用法。
- 遇到不同芯片差异时，不为了其他芯片破坏它的简单性和效率。
- 第一版优先面向 `TSSOP20` 封装的常用项目形态。

### 2.2 低成本兼容芯片

低成本兼容芯片指和 `STC8H1K08` 属于同一 STC8H 子系列，并且满足以下条件的型号：

- 使用相同或高度一致的 SFR/寄存器定义。
- 外设行为一致或差异可以通过编译期宏处理。
- 不需要运行期芯片识别。
- 不需要函数指针表、设备树或复杂适配层。
- 不会让主支持芯片生成额外代码。

这类芯片可以通过新增芯片配置头文件支持。

典型目标：

- `STC8H1K17`

是否正式列入支持，需要以实际编译和硬件验证为准。

### 2.3 暂不承诺支持的芯片

以下情况暂不承诺支持：

- 非 STC8H 系列芯片。
- STC8H 中寄存器模型明显不同的子系列。
- 外设数量、寄存器布局、中断向量或复用规则差异很大，需要明显增加抽象层的型号。
- 需要为兼容而增加运行期开销的型号。

如果以后实际项目遇到这些芯片，再按真实需求评估是否新增独立适配。

## 3. 芯片差异处理方式

允许的方式：

- 编译期宏。
- 独立芯片配置头文件。
- 只在需要的模块中处理局部差异。
- 示例工程选择对应芯片配置。

不允许的方式：

- 运行期判断芯片型号。
- 为所有芯片保留统一外设对象表。
- 未使用模块也参与初始化或注册。
- 为了兼容少用型号，让常用型号多占 ROM/RAM。

## 4. 第一版结论

第一版正式目标：

```text
STC8H1K08
```

第一版可评估兼容：

```text
STC8H1K17
```

其他芯片不预先设计支持。以后遇到真实项目需要时，再根据寄存器和外设差异决定是否加入。

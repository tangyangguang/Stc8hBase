# 开发工具链

## 1. 推荐工作流

如果主要开发环境是 macOS，推荐使用：

```text
VS Code / Cursor
  + SDCC
  + PlatformIO 或 Makefile
  + stcgal
```

其中：

- SDCC 负责 8051/STC8H C 代码编译。
- PlatformIO 或 Makefile 负责编译组织、示例工程和自动化检查。
- stcgal 负责命令行烧录 STC 芯片。

第一版以 SDCC + Makefile/PlatformIO 作为新项目主工作流。

老项目仍然允许使用 Keil C51。

## 2. SDCC 和 Keil C51 对比

| 项目 | SDCC | Keil C51 |
| --- | --- | --- |
| macOS 原生开发 | 支持 | 不适合 |
| 开源和自动化 | 适合 | 较弱 |
| Makefile/CI | 适合 | 不方便 |
| 老 8051 项目生态 | 一般 | 很成熟 |
| 极限代码体积优化 | 需要实测 | 通常较强 |
| STC 示例兼容 | 需要适配 | 很多示例天然面向 Keil |
| 长期可维护性 | 适合公共库和自动化 | 适合旧工程延续 |

结论：

- 新项目优先 SDCC，适合 macOS、自动化和公共库开发。
- 老项目不强制迁移，继续 Keil C51 更稳。
- 基础库源码必须同时兼容两者，不能写成 SDCC 专用。

## 3. 迁移策略

老项目通常不大，如果基础库效果好，可以接受以后用新方式重写。

推荐节奏：

1. 老项目先不动工具链，只按需接入基础库模块。
2. 新项目使用 SDCC + Makefile/PlatformIO。
3. 基础库持续保持 Keil/SDCC 兼容。
4. 老项目需要较大改造时，再评估是否用新方式重写。

## 4. 老项目兼容策略

老项目如果原来使用 Keil C51，可以接入本基础库，但基础库必须从第一版开始保持“编译器差异隔离”。

做法：

- 在 `core/stc8h_compiler.h` 中统一处理 `sfr`、`sbit`、`bit`、`code`、`xdata`、`interrupt` 等关键字差异。
- 同时处理 `data`、`idata`、`pdata`、`interrupt using`、`nop` 和 xdata SFR 访问差异。
- 不在业务模块中直接使用某个编译器专属写法。
- STC8H 寄存器定义集中放在芯片适配头文件或编译器适配层中。
- 芯片 SFR 定义只能有一个来源，优先使用 STC 官方或项目确认的芯片头文件，避免和本库重复定义。
- 公共 API 不暴露编译器专属 `bit` 返回值。
- 字符串参数默认按 RAM 指针处理，`code` 字符串另行提供专用接口。
- ISR 声明统一使用 `STC8H_INTERRUPT(name, vector)` 或 `STC8H_INTERRUPT_USING(name, vector, reg_bank)`。
- 示例可以优先支持 SDCC，但核心源码不能被 SDCC 绑定死。
- 如果老项目暂时继续用 Keil，只把需要的 `.c/.h` 模块加入 Keil 工程即可。
- 每个已实现模块都需要保持 Keil C51 可编译；如果某模块暂未完成 Keil 验证，必须在测试记录中标记。

## 5. 是否建议把老项目全部改成 SDCC

不建议一次性强制迁移。

更稳妥的方式是：

1. 新项目优先使用 macOS + SDCC 工作流。
2. 老项目如果还在维护，先按模块引入基础库。
3. 基础库源码同时保持 SDCC/Keil 可编译。
4. 等某个老项目需要较大改造时，再决定是否整体迁移到 SDCC。

## 6. 对基础库的要求

为了同时支持 macOS 日常开发和老项目改造，基础库需要遵守：

- 公共 API 使用普通 C 风格，不依赖编译器扩展。
- 编译器差异只出现在 `core` 层。
- 每个模块独立 `.h/.c`，方便 Keil 工程按需添加。
- 不依赖 PlatformIO 专属目录结构。
- 不依赖动态链接、系统库或宿主机能力。
- 示例工程可以多种形式存在，但库源码保持一致。

## 7. VS Code 和 SDCC 语法提示

SDCC 的 8051 扩展语法可能导致 VS Code C/C++ 插件误报。

第一版文档和示例需要提供编辑器配置说明：

- 添加 SDCC include path。
- 添加用于编辑器识别的语法兼容宏。
- 明确编辑器红线不等于 SDCC 编译失败。

这类配置只服务开发体验，不应影响固件编译结果。
